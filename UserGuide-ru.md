# Руководство пользователя Consul Viewer TUI

## 1. Назначение

`Consul Viewer TUI` — это консольное приложение для read-only просмотра состояния кластера Consul.

Приложение позволяет:

- просматривать основные данные Consul без каких-либо операций изменения;
- перемещаться по разделам, спискам и деталям с клавиатуры;
- анализировать сервисы, ноды, KV, ACL, сессии и метрики;
- применять фильтрацию и сортировку к текущему списку;
- просматривать полные JSON/текстовые значения во viewer-окне.

Приложение не выполняет `PUT`, `POST`, `DELETE` и не изменяет данные в Consul.

## 2. Запуск

Базовый запуск:

```bash
python consul-viewer.py
```

Основные параметры командной строки:

- `--addr` — адрес Consul HTTP API
- `--token` — ACL token
- `--refresh` — интервал автообновления в секундах
- `--timeout` — timeout сетевых запросов
- `--insecure` — отключить TLS verification
- `--dc` — явно указать datacenter
- `--ca-file` — путь к CA bundle
- `--cert-file` — путь к клиентскому сертификату
- `--key-file` — путь к приватному ключу клиента

Также поддерживаются переменные окружения:

- `CONSUL_HTTP_ADDR`
- `CONSUL_HTTP_TOKEN`

Примеры:

```bash
python consul-viewer.py --addr http://127.0.0.1:8500
python consul-viewer.py --addr https://consul.example.org:8501 --token <TOKEN>
python consul-viewer.py --refresh 10 --timeout 15
```

## 3. Общая компоновка интерфейса

Экран состоит из следующих областей:

- верхняя строка заголовка: datacenter, leader, адрес Consul и режим авторизации;
- строка вкладок (tab bar): разделы приложения;
- левая панель `Items`: список объектов текущего раздела;
- правая панель `Details`: детали по выбранному объекту;
- нижняя строка статуса: загрузка, stale/fresh, фильтры, сортировка, ошибки;
- нижняя строка подсказок клавиш.

Во многих сценариях используются popup-окна:

- help;
- viewer;
- ввод фильтра;
- выбор status filter;
- выбор instance filter;
- выбор сортировки;
- выбор сбрасываемых фильтров;
- ошибки;
- подтверждение выхода.

## 4. Разделы

### 4.1 Dashboard

Показывает краткое состояние локального Consul-агента и кластера:

- datacenter;
- локальный node;
- версию Consul;
- leader;
- количество peers;
- количество членов в local/remote DC;
- агрегированное `Agent state`.

В `Details` дополнительно выводятся:

- причины текущего `Agent state`;
- raft/serf telemetry;
- список членов кластера.

### 4.2 Telemetry

Раздел для просмотра метрик из:

- `/v1/agent/metrics?format=prometheus`

Показывает счетчики и индикаторы нагрузки:

- общий `Cluster state`;
- `Open FDs`;
- активные gRPC-соединения;
- `Blocking RPC`;
- `RPC errors`;
- `Raft main saturation`;
- `Raft FSM saturation`;
- количество потоков;
- потребление памяти;
- агрегированные счетчики объектов кластера.

Для метрик, где известен лимит, показывается:

- текущее значение;
- лимит;
- процент использования;
- цветовая оценка (`passing`, `warning`, `critical`).

Пример: `Open FDs` рассчитывается как `process_open_fds / process_max_fds`.

### 4.3 Services

Показывает агрегированный список сервисов:

- имя сервиса;
- теги;
- число инстансов;
- `Status` в формате `N/M`, где `N` — количество passing-инстансов, `M` — общее число инстансов.

При выборе сервиса в `Details` выводится:

- summary по health;
- статус сервиса;
- список инстансов;
- checks по каждому инстансу.

`Enter` на сервисе открывает список его инстансов.

### 4.4 Services / Instances

В режиме инстансов показывается список инстансов выбранного сервиса:

- instance ID;
- адрес;
- порт;
- `Status` в формате `N/M`, где `N` — число passing-checks, `M` — общее число checks.

В `Details` для инстанса показываются:

- service / service ID;
- node;
- адрес и порт;
- статус;
- теги;
- service meta;
- node meta;
- список checks.

### 4.5 Nodes

Показывает список нод:

- имя;
- адрес;
- datacenter;
- `Status` в формате `N/M`, где `N` — количество passing-инстансов на ноде, `M` — общее число инстансов.

Важно:

- статус строки ноды определяется по node checks, а не по состоянию сервисов на ней;
- состояние сервисов выводится отдельно в деталях.

В `Details` для ноды выводятся:

- адрес;
- datacenter;
- meta;
- `Node status`;
- `Services status`;
- список инстансов;
- node checks;
- service checks.

`Enter` на ноде открывает список инстансов, зарегистрированных на этой ноде.

### 4.6 Nodes / Instances

В режиме инстансов ноды показываются:

- instance ID;
- адрес;
- порт;
- `Status` в формате `N/M` по checks.

В `Details` выводятся:

- сервис;
- service ID;
- node;
- адрес и порт;
- статус;
- теги;
- service meta;
- node meta;
- список checks.

`Enter` на инстансе ноды:

- переключает в раздел `Services`;
- накладывает точный фильтр по имени соответствующего сервиса;
- сразу выбирает этот сервис;
- сразу загружает его детали.

### 4.7 KV

Раздел для read-only просмотра KV.

Показывает:

- директории и ключи по текущему prefix;
- переход внутрь prefix;
- preview значения;
- metadata ключа;
- полный просмотр значения через viewer.

Поддерживаются:

- переход в директорию;
- возврат к родительскому prefix;
- просмотр текстовых и бинарных значений.

### 4.8 Sessions

Показывает список активных сессий:

- name;
- node;
- TTL;
- behavior.

В `Details` доступны подробности выбранной сессии.

### 4.9 Tokens

Показывает ACL tokens, если ACL endpoints доступны и токен пользователя имеет права на чтение.

В списке показываются:

- последний блок `AccessorID`;
- `Description`.

`SecretID` в обычном списке и деталях не показывается автоматически.

Дополнительно:

- `F4` открывает popup с `SecretID` выбранного токена;
- `Enter` открывает список связанных ролей и политик.

### 4.10 Policies

Показывает ACL policies:

- имя policy;
- description.

В `Details` доступны:

- `PolicyID`;
- description;
- `Rules`;
- список ролей;
- таблица токенов, где политика используется.

Для таблицы токенов показываются:

- `AccessorID` (последний блок);
- `Scope`;
- `Source` (`direct`, `via role`, `direct+role`);
- `Description`.

### 4.11 Roles

Показывает ACL roles:

- имя role;
- description;
- список policies.

В `Details` доступны:

- `RoleID`;
- description;
- список policies;
- таблица токенов, где роль используется.

`Enter` открывает список связанных policies.

### 4.12 Auth

Показывает ACL auth methods:

- `Name`;
- `Type`;
- `Locality`.

В `Details` выводятся:

- базовые поля auth method;
- `TokenLocality`;
- специфичные поля конфигурации;
- `Bound audiences`;
- JWT/JWKS-related поля;
- поля для `kubernetes` auth method;
- дополнительные поля `Config`, если они есть.

### 4.13 Mesh

Раздел присутствует как placeholder, но в текущей версии не реализован.

## 5. Основная навигация

### 5.1 Перемещение по разделам

- `Tab` — следующий раздел
- `Shift+Tab` — предыдущий раздел
- `Ctrl+Tab` — предыдущий раздел
- `0..9` — переход в numbered section
- `T` — быстрый переход в `Telemetry`

Соответствие numbered sections:

- `0` — `Auth`
- `1` — `Dashboard`
- `2` — `Services`
- `3` — `Nodes`
- `4` — `KV`
- `5` — `Sessions`
- `6` — `Tokens`
- `7` — `Policies`
- `8` — `Roles`
- `9` — `Mesh`

### 5.2 Переключение между панелями

- `Left` / `Right` — переключение фокуса между `Items` и `Details`

### 5.3 Перемещение по списку

- `Up` / `Down` — перемещение по строкам списка
- `PgUp` / `PgDn`, `Home`, `End` — стандартное поведение `urwid` для длинных списков

### 5.4 Drill-down и возврат

- `Enter` — открыть выбранный объект или перейти глубже
- `Backspace` — вернуться назад

Типовые сценарии:

- `Services -> Enter` — открыть инстансы сервиса
- `Nodes -> Enter` — открыть инстансы ноды
- `Node Instance -> Enter` — перейти в `Services` с фильтром по соответствующему сервису
- `Tokens -> Enter` — открыть связанные роли и policies
- `Roles -> Enter` — открыть связанные policies
- `KV directory -> Enter` — перейти в prefix

## 6. Просмотр данных и popup-окна

### 6.1 Help

- `F1` — открыть встроенную справку

### 6.2 Full Viewer

- `F3` — открыть full viewer для текущего объекта

Примеры:

- для KV — полный текст значения или preview;
- для JSON-разделов — полный JSON;
- для `Telemetry` — полный Prometheus payload.

### 6.3 SecretID токена

- `F4` — показать `SecretID` выбранного ACL token

Если выбран не token или у token нет `SecretID`, приложение покажет сообщение в status bar.

### 6.4 Подтверждение выхода

- `F10` или `Esc` на основном экране открывают диалог `Confirm Exit`
- `Yes` выбран по умолчанию
- `Enter` подтверждает выбранный вариант
- `Y` — быстрый выход
- `N`, `Esc`, `F10` — отмена выхода

## 7. Фильтрация

Фильтры применяются последовательно:

1. текстовый фильтр;
2. status filter;
3. structured instance filter;
4. затем сортировка.

### 7.1 Текстовый фильтр

- `F7` или `/`

Открывает popup для ввода текстового фильтра по текущему списку.

Фильтр ищет подстроку по значениям строки.

Специальный случай:

- в `Services` используется внутренний exact-filter вида `=service-name` для перехода из `Node Instance -> Services`

Этот exact-filter нужен для навигации и не должен ломать просмотр списка инстансов сервиса.

### 7.2 Status Filter

- `F6`

Доступен для:

- списка сервисов;
- списка нод;
- списков инстансов.

Варианты:

- `Passed`
- `Warning`
- `Critical`
- `No checks`

Можно выбрать любую комбинацию. Логика внутри фильтра — `OR`.

Примеры:

- `Passed + Warning` — показать строки со статусом `passing` или `warning`
- только `Critical` — показать только проблемные объекты

Состояние status filter хранится отдельно для:

- `Services`
- `Nodes`
- `Instances`

### 7.3 Instance Filter

- `F9`

Доступен только в списках инстансов.

Позволяет фильтровать по:

- наличию тегов;
- отсутствию тегов;
- наличию meta keys;
- отсутствию meta keys;
- regex по имени meta key;
- regex по значению meta;
- чувствительности к регистру;
- режиму regex/substring.

Поля:

- `Has tags`
- `No tags`
- `Has meta keys`
- `No meta keys`
- `Meta key regex`
- `Meta value regex`
- `Case sensitive`
- `Regex enabled`

Правила:

- списки тегов и ключей вводятся через запятую;
- `Has ...` означает, что все перечисленные элементы должны присутствовать;
- `No ...` означает, что перечисленные элементы должны отсутствовать;
- если заданы regex для key и value, должна найтись хотя бы одна подходящая пара `key/value`;
- если regex некорректен, используется безопасный substring fallback.

### 7.4 Сброс фильтров

- `F8`

Открывает popup `Clear Filters`, где можно выбрать, какие фильтры сбросить:

- `Text filter`
- `Status filter`
- `Instance filter`

По умолчанию все чекбоксы включены.

Это позволяет сбрасывать только нужную часть фильтрации, не затрагивая остальные настройки.

## 8. Сортировка

- `F11`

Открывает popup `Sort` для текущего списка.

Можно выбрать:

- поле сортировки;
- направление `Descending`;
- `Default order`, чтобы снять пользовательскую сортировку.

Сортировка хранится отдельно для каждого вида списка.

Примеры:

- сортировка `Services` по имени;
- сортировка `Services / instances` по адресу;
- сортировка `Telemetry` по `Usage %`;
- сортировка `Tokens links` по типу или имени.

## 9. Переходы между связанными объектами

### 9.1 Из инстанса к ноде

- `Ctrl+N`

Работает в списках инстансов.

Переключает приложение в раздел `Nodes` и выбирает соответствующую ноду.

### 9.2 Из инстанса к сервису

- `Alt+S`

Работает в списках инстансов.

Переключает приложение в раздел `Services` и выбирает соответствующий сервис.

### 9.3 Из инстанса ноды к отфильтрованному сервису

- `Enter` в списке `Nodes / Instances`

Переключает в `Services` и накладывает точный фильтр по имени сервиса выбранного инстанса.

## 10. Цветовая индикация

Используется палитра в стиле Far Manager:

- темно-синий фон;
- светлый основной текст;
- контрастные активные элементы;
- серые popup-окна с черным текстом.

Основная цветовая логика:

- `passing` — зеленый
- `warning` — желтый
- `critical` — красный

Подсветка применяется:

- к строкам списков;
- к статусным строкам в `Details`;
- к checks в деталях сервисов и инстансов;
- к `Agent state` и telemetry-показателям.

## 11. Строка статуса

Нижняя строка статуса показывает:

- текущий раздел;
- состояние `loading/idle`;
- состояние `fresh/stale`;
- активный текстовый фильтр;
- активный status filter;
- active instance filter summary;
- активную сортировку;
- путь в `KV`;
- последнее сообщение;
- последнюю ошибку.

Это основной диагностический канал приложения.

## 12. Обработка ошибок и degraded mode

Если endpoint недоступен или вернул ошибку:

- приложение не очищает последний корректный snapshot;
- данные остаются на экране;
- раздел помечается как `stale`;
- ошибка выводится в status bar;
- также может открыться popup с текстом ошибки.

Для ACL:

- если ACL endpoints недоступны, соответствующие разделы остаются видимыми, но недоступными;
- при попытке открыть их пользователь получает понятное сообщение.

## 13. Ограничения текущей версии

- раздел `Mesh` пока не реализован;
- часть оценок состояния (`Agent state`, telemetry thresholds) является эвристикой;
- для некоторых метрик состояние вычисляется по локально заданным порогам, а не по “официальному” статусу Consul;
- приложение ориентировано на keyboard-first работу и read-only диагностику.

## 14. Практические сценарии

### 14.1 Найти проблемный сервис

1. Перейти в `Services`
2. Нажать `F6`
3. Оставить только `Critical`
4. Выбрать сервис
5. Нажать `Enter` для просмотра инстансов
6. В `Details` посмотреть checks

### 14.2 Найти инстансы по тегу и meta

1. Перейти в `Services`
2. Нажать `Enter` на нужном сервисе
3. Нажать `F9`
4. Заполнить `Has tags`
5. При необходимости добавить `Has meta keys` или regex
6. Нажать `Enter`

### 14.3 Проверить риск исчерпания дескрипторов

1. Перейти в `Telemetry` клавишей `T`
2. Выбрать `Open FDs`
3. Посмотреть:
   - текущее количество открытых дескрипторов;
   - лимит;
   - процент использования;
   - цветовой статус

### 14.4 Быстро вернуться назад

1. Уйти вглубь: `Nodes -> Instances -> Service`
2. Использовать `Backspace` для пошагового возврата

## 15. Рекомендации по использованию

- используйте `F6` для быстрого выделения проблемных объектов;
- используйте `F9`, когда текстового фильтра недостаточно;
- используйте `F11` для сортировки больших списков;
- используйте `F3`, когда нужен полный payload без сокращений;
- в `Telemetry` ориентируйтесь в первую очередь на:
  - `Open FDs`
  - `RPC errors`
  - `Blocking RPC`
  - `Raft saturation`
